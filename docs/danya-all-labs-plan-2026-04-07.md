# План закрытия всех лаб `people/danya` + отчеты (вариант 19)

## Summary
- Цель: выполнить и довести до готовности все существующие лабы в `people/danya`, а также создать отсутствующие лабы, для которых есть подтвержденные задания в `subjects/*`; подготовить отчет `DOCX` к каждой лабе.
- Workflow (обязательно по правилам репо): для каждого блока последовательно применяем `design-feature.md` -> `implement-feature.md`; `prepare-release.md` не применяем, если изменения не релизного уровня.
- Критерий готовности: у каждой лабы есть рабочий код/проект, инструкции запуска, проверка выполнения и отчет в `DOCX`.

## Implementation Changes
- Этап 1. Аудит и матрица покрытия.
- Собрать единый трекер "лаба -> статус кода -> статус отчета -> источник задания -> команда проверки".
- Зафиксировать риск/rollback по каждому блоку (`low/medium/high`) до правок.
- Синхронизировать `subjects/people-variants.md`: `program-interface = 19` для `danya`.
- Этап 2. Существующие лабы (доведение до сдачи).
- `lab-1..lab-8`: проверить соответствие варианту 19, запуск, исправить дефекты, обновить README при изменении поведения.
- `distributed-processing/lab-1-threads/Variant19`: проверить обе задачи (потоки + producer/consumer), воспроизводимость запуска, обновить отчет при расхождениях.
- `web/*` (lab1/3/5/7): проверить работоспособность страниц/скриптов/php, добавить недостающие инструкции и отчеты по требованиям.
- Этап 3. Отсутствующие лабы (только где есть подтвержденное ТЗ).
- Реализовать недостающие лабы из `subjects/*`, где задание извлекается однозначно (включая `distributed-processing/lab-02`).
- Для PI 9-11: не реализовывать до получения ТЗ/методичек (решение зафиксировано).
- Этап 4. Отчеты `DOCX` к каждой лабе.
- Политика: обновить существующие отчеты (`docs/laba-1..6`) + добавить недостающие.
- Для каждой лабы: цель, постановка, вариант, листинг ключевого кода, скриншоты/результаты, выводы, команды запуска.
- Унифицировать структуру и стиль отчетов, не ломая уже принятые преподавателем части без необходимости.
- Этап 5. Документация и контроль качества.
- Обновить релевантные README и чеклисты после каждой существенной правки.
- Выполнить scope-check, docs-sync-check, test/build-check, rollback-feasibility-check для каждого затронутого модуля.

## Public Interfaces / Contracts
- Сохраняем обратную совместимость существующих контрактов:
- форматы `conf.prop` и ожидаемые ключи подключения к БД;
- команды запуска (`run-lab8.cmd`, `run-lab8.ps1`, Maven/Java команды);
- структура существующих директорий и артефактов студента.
- Новые публичные интерфейсы допускаются только для новых отсутствующих лаб, с обязательной фиксацией в README этих лаб.

## Test Plan
- Java Core (lab1-5): компиляция `javac`, запуск main-классов, smoke-проверка основных сценариев и валидации.
- JavaFX/Maven (lab6-8): `mvn -q -DskipTests compile` + smoke-run ключевых экранов/действий.
- JDBC (lab7-8): проверка чтения `conf.prop`, соединения, минимум одного рабочего запроса/операции.
- Distributed-processing:
- lab1: корректность синхронизации потоков, отсутствие гонок в базовых сценариях;
- lab2 (если добавляется): обмен по сокетам/каналам по методичке, корректный happy-path + ошибка соединения.
- Web: синтаксическая/базовая runtime-проверка HTML/CSS/JS/PHP и ручной smoke по пользовательскому сценарию.
- Отчеты: для каждой лабы проверить наличие DOCX, полноту разделов, соответствие фактическому коду/скриншотам.

## Assumptions (Locked)
- Вариант `19` используется для `program-interface` и `distributed-processing`.
- Формат отчетов: только `DOCX`.
- Политика отчетов: обновить существующие + добавить недостающие.
- Если для PI 9-11 нет методичек/ТЗ в репо, работы по ним не выполняются до получения исходных требований.
